# PR Comment Scorer

PRレビューコメントを修正する前にスコアリングし、対応要否を判定するトリアージスキル。

## 重要な原則

**コメントを見たら即座に修正しようとしない。** まず評価し、対応価値のあるものだけ対応する。

## ワークフロー

### 1. コメント収集

```bash
# 現在のブランチのPRを特定
gh pr list --head $(git branch --show-current) --json number,title,url --jq '.[0]'

# レビューコメント（コード上のコメント）を取得
gh api repos/{owner}/{repo}/pulls/{number}/comments

# イシューコメント（PR全体へのコメント）を取得
gh api repos/{owner}/{repo}/issues/{number}/comments
```

### 2. コメントのグルーピング

同一内容のコメントが複数ファイルにある場合（Copilot等の自動レビューに多い）、1グループとしてまとめて評価する。

### 3. スコアリング

各コメント/グループに対して以下の軸で 1-5 のスコアを付ける。

| 軸 | 1 (低) | 5 (高) |
|----|--------|--------|
| **技術的妥当性** | 指摘内容が誤りまたは的外れ | 技術的に正確で根拠がある |
| **影響度** | 変えても実質的な改善なし | 品質・安全性・保守性に大きく寄与 |
| **対応優先度** | マージをブロックする理由なし | 未対応ならマージすべきでない |
| **false positive度** | コンテキストを正しく理解した指摘 | 文脈を無視した汎用的な指摘 |

#### スコアリング時の判断基準

- **コメント元を考慮する**: 人間のレビュアー > 自動レビューツール（Copilot等）。自動ツールはコンテキストを読まずにパターンマッチで指摘する傾向がある
- **提案の実効性を検証する**: 提案された変更を実際に適用した場合に改善があるか、コードを読んで確認する
- **変更対象のスコープを考慮する**: テストコード vs 本番コード、ホットパス vs 初期化処理など
- **PRの目的との整合性**: PRの変更意図と関係ない改善提案はスコープ外として扱う

### 4. 出力

以下の形式で結果を出力する。

#### コメントが複数ある場合

グループ/コメントごとにスコア表を出力:

```
## PR #{number} コメントスコアリング

### コメント1: {コメントの要約} (by {user})
対象: {ファイルパス} (他N件)

> {コメント本文の引用}

| 観点 | スコア (1-5) | 理由 |
|------|-------------|------|
| 技術的妥当性 | X | ... |
| 影響度 | X | ... |
| 対応優先度 | X | ... |
| false positive度 | X | ... |

**総合スコア: X/5 — {対応推奨 / 対応任意 / 対応不要}**
{判断の根拠を1-2文で}
```

#### 総合判定の基準

- **4-5: 対応推奨** — 技術的に正しく、改善効果が高い
- **3: 対応任意** — 正しいが影響が限定的。時間があれば対応
- **1-2: 対応不要** — false positiveまたは実効性なし

### 5. 次のアクション

スコアリング結果を提示した後、ユーザーの判断を待つ。自動で修正を開始しない。
